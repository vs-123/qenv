* QENV

*qenv* is a simple, POSIX-compliant, zero-dependency environment variable inspection utility tool designed for quick and memory-efficient filtering across diverse shell environments.

** BUILD INSTRUCTIONS

Obtain a local copy of this repository with =git clone= and enter it.
#+begin_example
  % git clone https://github.com/vs-123/qenv.git
  % cd qenv
#+end_example

Create a =build= directory and build this project with =cmake= inside it
#+begin_example
  % mkdir build && cd build
  % cmake ..
  % cmake --build .
  % ls -F
  qenv*  compile_commands.json  Makefile  CMakeCache.txt
#+end_example

You may now use the generated binary =qenv=
#+begin_example  
  % ./qenv --help
#+end_example

** FEATURES

- **[STREAM-ORIENTED TRAVERSAL]** -- Direct iteration through the global environment pointer array. This ensures constant memory usage regardless of the environment's size. This satisfies the requirement to handle multi-megabyte environment blocks without buffering the entire output set.
  
- **[PORTABLE MEMORY CALCULATION]** -- Flag =-v= is available to provide a total environment block size measurement by aggregating the length of all environment strings and the pointer array overhead. This maintainins POSIX portability by avoiding non-standard filesystem interfaces.

*** OBSERVATION

Notice the difference in maximum resident set size. The observed =80= KB difference in maximum RSS between =qenv= (=1296= KB) and the system-native =env= (=1216= KB) is primarily attributed to dynamic loading and runtime initialisation overhead.

Even though =qenv= is architecturally lean, the system =env= utility is frequently pre-linked and cached as part of the core system runtime. Thus, the remaining =1296= KB represents the functional "floor" for an independent C process under glibc initialisation.

This baseline encompasses the minimal stack, heap and data segments that are required to bootstrap the process and execute the POSIX =write()= interface. Hence, this renders it the lowest achievable footprint for this implementation without moving to non-standard, minimal-runtime alternatives like =dietlibc=, etc.
